一、技术领域

本发明涉及数据工程与知识图谱技术领域，具体涉及一种将来自多种异构数据源的原始数据，通过统一适配器接口与声明式映射配置，自动转化为语义本体图谱（Ontology Graph）的方法、系统及计算机可读存储介质。

二、背景技术

随着企业信息化建设的深入推进，企业内部普遍存在大量异构数据源，包括关系型数据库、结构化文件、消息队列及对外服务接口等。上述数据源在建设时通常相互独立，数据格式、字段命名及语义表达各异，形成严重的数据孤岛现象。

现有技术通常采用抽取—转换—加载（ETL）管道或数据仓库方案对上述异构数据进行整合。然而，上述现有技术方案存在以下缺陷：

缺陷一、集成复杂度随系统数量呈平方级增长。当企业存在 N 个异构数据系统时，点对点集成模式需维护最多 N(N-1)/2 条独立的集成管道。当任意数据系统的字段定义发生变更时，所有依赖该字段的下游管道均需人工修复，维护代价极高。

缺陷二、缺乏统一的跨系统语义关联机制。相同的业务概念在不同系统中往往以不同的字段名称存储，现有方案无法自动建立跨系统的语义对齐，导致数据间的关联关系需人工维护，准确性低、一致性差。

缺陷三、无法支持人工智能系统的直接查询。由于缺乏统一的语义表示层，现有方案中的人工智能 Agent 无法直接理解业务数据的概念关系，必须针对每个数据系统单独开发适配接口，工程成本高，且难以实现跨系统的自然语言查询与智能推理。

有鉴于此，本领域亟需一种能够将多种异构数据源统一接入、自动建立语义关联、并天然支持人工智能查询的数据转化方法及系统。

三、发明内容

本发明的目的在于克服现有技术的上述缺陷，提供一种将异构原始数据转化为统一语义本体图谱的方法及系统，以将多源异构数据的集成复杂度从 O(N²) 降低至 O(N)，并建立可被人工智能系统直接使用的结构化语义图谱。

为实现上述目的，本发明采用如下技术方案：

本发明提供一种将异构原始数据转化为语义本体图谱的方法，包括以下步骤：步骤 S1，通过统一适配器接口接收来自至少一个异构数据源的原始数据；步骤 S2，根据声明式映射配置规则将所述原始数据映射为本体对象类型及其属性；步骤 S3，依据所述映射规则中声明的字段关联关系自动推导本体对象间的语义链接类型；步骤 S4，将转化结果以事件流方式写入语义图谱存储层。

本发明还提供一种实现上述方法的系统，以及一种存储有计算机程序的计算机可读存储介质，所述程序被处理器执行时实现上述方法的各步骤。

有益效果

与现有技术相比，本发明具有以下有益效果：

一、降低集成复杂度。采用统一适配器接口，N 个异构数据源仅需实现 N 个适配器，将集成管道数量从最多 N(N-1)/2 降低至 N，实现复杂度从 O(N²) 至 O(N) 的本质性改善。

二、自动建立语义关联。通过解析声明式映射配置中的外键关联声明及数据源 Schema 约束，自动推导本体对象间的语义链接类型，无需人工逐一定义关联关系。

三、原生支持人工智能查询。所构建的语义本体图谱天然作为人工智能 Agent 的知识库，Agent 可通过结构化的对象集合接口（ObjectSet API）以自然语言发起跨实体的图遍历查询，无需针对每个数据源单独开发适配接口。

四、支持增量实时同步。适配器接口内置增量游标机制，支持仅传输发生变更的数据记录，事件流架构保证图谱数据的最终一致性，无需全量重建。

五、零代码数据接入。声明式映射配置文件支持由用户界面向导自动生成，非工程技术人员亦可独立完成异构数据源的接入配置。

实施例1 — 统一适配器接口定义及声明式映射配置文件的具体实现

下面结合附图及实施例对本发明作进一步的详细说明。以下实施例用于说明本发明，但不用来限制本发明的范围。

实施例1 以关系型数据库及结构化 CSV 文件为例，说明统一适配器接口（SourceAdapter）及声明式映射配置文件（TomlMapping）的具体实现方式。

如图1所示，所述统一适配器接口以异步流式方式输出标准化记录（CanonicalRecord）。每条标准化记录包含数据源标识符、记录唯一标识符、字段键值映射表及时间戳信息。适配器接口接受可选的游标参数（Cursor），当游标参数不为空时，接口仅输出自该游标时间点之后发生变更的数据记录，实现增量同步；当游标参数为空时，接口输出数据源全量记录。

清单1 — 统一适配器接口及标准化记录类型定义（Rust 语言）
// adapters.rs use async_trait::async_trait; use futures_core::Stream; #[derive(Debug, Clone)] pub struct SourceDescriptor { pub id: String, pub has_cursor: bool, pub partitions: Option<u32>, } #[async_trait] pub trait SourceAdapter: Send + Sync { fn id(&self) -> &str; async fn describe(&self) -> SourceDescriptor; fn stream( &self, since: Option<Cursor>, ) -> Box<dyn Stream<Item = Result<CanonicalRecord, AdapterError>> + Unpin + Send>; } // model.rs — 标准化记录（所有数据源的统一输出格式） #[derive(Debug, Clone, Serialize, Deserialize)] pub struct CanonicalRecord { pub source: SourceId, // 适配器标识 pub ns: String, // 命名空间（用于映射过滤） pub schema: SchemaId, // Schema 版本标识 pub payload: serde_json::Value, // 原始字段键值对 pub ts: OffsetDateTime, // 采集时间戳 pub cursor: Option<Cursor>, // 增量游标 }

所述声明式映射配置文件（TomlMapping）采用结构化文本格式，包含数据源连接信息、目标本体对象类型声明、字段映射规则列表及外键关联关系声明。该配置文件可由用户界面向导根据数据源 Schema 自动生成，亦可由开发人员手动编写。以员工数据 CSV 文件的接入配置为例，具体格式如下所示。

清单2 — 声明式映射配置文件示例（TOML 格式）
# Employee 映射配置（TOML） version = "v1" entity = "Employee" [from] ns = "csv.employees" # 仅处理该命名空间的 CanonicalRecord [id] field = "id" # 主键字段声明 # 字段映射规则：target = "source_field|cast_type" [map] name = "name|str" department = "department|str" level = "level|str" salary = "salary|float" # 自动类型转换 # 外键关联声明 → 运行时自动推导 Link 事件 [[links]] rel = "BelongsTo" to_entity = "Department" from_key = "id" # 当前对象主键 to_key = "dept_id" # 关联目标对象 id

转化引擎解析上述配置后，对每条 CanonicalRecord 执行字段映射，生成 Employee 类型本体对象实例（OntologyEvent::Upsert），并依据 [[links]] 声明，自动推导 Employee → BelongsTo → Department 语义关联（OntologyEvent::Link），无需修改任何核心代码。

实施例2 — 本体事件流的生成与写入、及人工智能 Agent 查询接口的具体实现

实施例2 说明转化引擎输出本体事件流（OntologyEvent Stream）、写入图谱存储层，以及人工智能 Agent 通过对象集合接口（ObjectSet API）查询语义图谱的具体实现方式。

所述本体事件流包含三种变体：Upsert（对象新增/更新）、Delete（对象删除）及 Link（语义关联建立）。事件类型的完整定义如清单3所示。本体管理器（OntologyManager）对每个适配器并行启动 tokio::spawn 任务，以批量流式方式消费事件并写入持久化层，具体实现如清单4所示。

清单3 — 本体事件类型定义（Rust 语言）
// model.rs #[derive(Debug, Clone, Serialize, Deserialize)] pub enum OntologyEvent { /// 对象新增 / 更新（upsert 语义） Upsert { object: OntologyObject, }, /// 对象删除 Delete { id: OntologyId, }, /// 语义关联建立（由外键映射自动推导） Link { from: OntologyId, to: OntologyId, rel: String, attrs: Option<LinkAttrs>, time: TimeBounds, provenance: Provenance, }, } /// 本体对象实例 #[derive(Debug, Clone, Serialize, Deserialize)] pub struct OntologyObject { pub id: OntologyId, pub entity_type: String, pub attrs: BTreeMap<AttrId, Value>, pub time: TimeBounds, pub version: i64, pub provenance: Provenance, }
清单4 — 本体管理器核心写入逻辑（Rust 语言）
// manager.rs — 每个适配器并行 spawn 一个 tokio task impl<R: OntologyRepository + 'static> OntologyManager<R> { pub async fn run(&self, since: Option<Cursor>) -> Result<()> { let mut tasks = Vec::new(); for adapter in &self.adapters { let mapper = self.mappers .get(adapter.id()) .cloned() .expect("missing mapper"); let (repo, schema) = (self.repo.clone(), self.schema.clone()); let batch = self.batch_size; let task = tokio::spawn(async move { let mut stream = adapter.stream(since.clone()); let mut buf = Vec::with_capacity(batch); while let Some(item) = stream.next().await { let rec = item.map_err(|e| anyhow::anyhow!(e))?; let events = mapper.apply(&rec, &schema) .map_err(|e| anyhow::anyhow!(e))?; buf.extend(events); if buf.len() >= batch { repo.apply(&buf).await?; buf.clear(); } } if !buf.is_empty() { repo.apply(&buf).await?; } Ok::<(), anyhow::Error>(()) }); tasks.push(task); } for t in tasks { t.await??; } Ok(()) } }

清单5展示完整的入库流程调用示例（取自 examples/12_ingest_csv_ontology.rs）：构建 CsvAdapter 与 TomlMapping，注入 OntologyManager，调用 run(None) 执行全量入库；传入游标（Some(cursor)）则切换为增量同步模式，结果写入 InMemoryRepository 后可替换为任意持久化后端。

清单5 — OntologyManager 完整入库调用示例（examples/12_ingest_csv_ontology.rs）
// examples/12_ingest_csv_ontology.rs（精简） use palantir_ontology_manager::{ adapters_csv::CsvAdapter, manager::OntologyManager, mapping_toml::TomlMapping, repository::InMemoryRepository, }; #[tokio::main] async fn main() -> anyhow::Result<()> { let adapter: Arc<dyn SourceAdapter> = Arc::new( CsvAdapter::new("csv.employees", "data/employees.csv", "csv.employees", "employees_v1"), ); let mapping: Arc<dyn Mapping> = Arc::new( TomlMapping::from_str(EMPLOYEE_TOML)?, ); let repo = Arc::new(InMemoryRepository::new()); OntologyManager::new( vec![adapter], mappers, // HashMap<String, Arc<dyn Mapping>> repo.clone(), schema, ) .run(None) // None = 全量；Some(cursor) = 增量 .await?; println!("ingested: {}", repo.events.lock().await.len()); Ok(()) }

说明书摘要（全文）
本发明公开了一种将异构原始数据转化为统一语义本体图谱的方法、系统及计算机可读存储介质。所述方法包括：通过统一适配器接口以标准化记录流方式接收来自关系型数据库、结构化文件、消息队列等异构数据源的原始数据；根据声明式映射配置文件将所述记录的字段映射为本体对象类型及其属性；依据所述配置文件中声明的外键关联关系自动推导本体对象间的语义链接类型；将转化结果以事件流方式写入语义图谱存储层，构建语义本体图谱。

与现有技术相比，本发明将异构数据源的集成复杂度从O(N²)降低至O(N)；通过声明式映射配置实现零代码数据接入；利用游标机制支持增量实时同步；所构建的语义图谱天然支持人工智能 Agent 以自然语言执行跨实体图遍历查询。本发明适用于企业多源异构数据治理与人工智能辅助决策场景。
