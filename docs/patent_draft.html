权　利　要　求　书

  1. 一种将异构原始数据转化为语义本体图谱的方法，其特征在于，包括以下步骤：

    步骤 S1：通过统一适配器接口（SourceAdapter）接收来自至少一个异构数据源的原始数据，所述异构数据源包括但不限于本地文件系统、关系型数据库及 REST API，所述统一适配器接口以标准化记录流的形式输出所述原始数据；

    步骤 S2：根据预先存储的声明式映射配置，将所述标准化记录流中的每条记录转化为本体对象实例（ObjectInstance），所述本体对象实例包含对象标识符、对象类型标识符及属性键值映射；

    步骤 S3：根据所述声明式映射配置中声明的字段关联关系，自动推导所述本体对象实例之间的语义关联关系（LinkInstance），所述语义关联关系包含源对象标识符、目标对象标识符及关联类型标识符；

    步骤 S4：将所述本体对象实例及所述语义关联关系以事件流的形式写入本体事件总线（OntologyEventBus），所述事件流由本体管理器（OntologyManager）消费并持久化至本体知识库；

    步骤 S5：通过语义查询接口对所述本体知识库中的对象集合（ObjectSet）执行遍历、过滤及聚合操作，返回结构化查询结果。

  2. 一种将异构原始数据转化为语义本体图谱的系统，其特征在于，包括：

    数据适配层，包括针对不同异构数据源分别实现的多个适配器模块，各适配器模块实现统一适配器接口（SourceAdapter），用于将所述异构数据源的原始数据转化为标准化记录流；

    映射引擎，用于读取预先存储的声明式映射配置，并根据所述声明式映射配置将所述标准化记录流转化为本体对象实例及语义关联关系；

    本体事件总线，用于接收所述映射引擎输出的本体对象实例及语义关联关系，并以事件流的形式分发至订阅方；

    本体管理器，与所述本体事件总线连接，用于消费所述事件流并将所述本体对象实例及所述语义关联关系持久化至本体知识库；

    语义查询接口，与所述本体知识库连接，用于对所述本体知识库执行对象集合遍历、关系路径导航及属性过滤操作。

  3. 一种计算机可读存储介质，其上存储有计算机程序，其特征在于，所述计算机程序被处理器执行时实现权利要求1所述方法的步骤。

  4. 根据权利要求1所述的方法，其特征在于，所述步骤 S1 还包括：

    为每个适配器模块维护一个持久化游标（Cursor），所述游标记录上次同步的偏移量或时间戳；在每次同步时，所述适配器模块仅读取所述游标之后新增或更新的数据记录，并在同步完成后更新所述游标，以实现增量同步而非全量重导入。

  5. 根据权利要求1所述的方法，其特征在于，所述声明式映射配置以 TOML 格式存储，包含以下字段：数据源标识符、目标对象类型标识符、主键字段名称映射、属性字段名称映射及外键关联字段名称映射；所述映射引擎在运行时动态加载所述 TOML 配置文件，无需修改系统代码即可新增或调整数据源映射规则。

  6. 根据权利要求1所述的方法，其特征在于，所述步骤 S3 中自动推导语义关联关系的具体实现方式为：所述映射引擎读取所述声明式映射配置中的外键字段列表，对于每条本体对象实例记录，将外键字段值解析为目标对象标识符，并根据所述声明式映射配置中的关联类型标识符生成对应的语义关联关系实例，从而无需人工编写关联规则代码。

  7. 根据权利要求1所述的方法，其特征在于，所述步骤 S4 中写入本体事件总线的事件流中，每条事件包含以下类型之一：对象创建事件（ObjectCreated）、对象更新事件（ObjectUpdated）、对象删除事件（ObjectDeleted）、关联创建事件（LinkCreated）及关联删除事件（LinkDeleted）；所述本体管理器根据事件类型执行对应的持久化操作，以维护本体知识库的一致性。

  8. 根据权利要求2所述的系统，其特征在于，还包括向量索引模块；所述向量索引模块在所述本体对象实例写入本体知识库时，对指定属性字段的文本内容进行向量嵌入（Embedding）计算，并将所得向量存储至向量数据库；所述语义查询接口支持以自然语言输入为查询条件，通过所述向量数据库执行近似最近邻搜索得到候选对象集合，再结合图遍历操作精化查询结果。

  9. 根据权利要求2所述的系统，其特征在于，所述本体知识库支持多租户隔离模式，包括：共享表模式（Shared Schema），通过租户标识符列实现行级隔离；独立命名空间模式（Schema Per Tenant），为每个租户创建独立数据库命名空间；及物理隔离模式（Database Per Tenant），为每个租户提供独立数据库实例；所述系统在初始化时根据配置参数选择上述隔离模式之一并在整个运行周期内保持一致。

  10. 根据权利要求2所述的系统，其特征在于，还包括本体模式注册表（SchemaRegistry）；所述本体模式注册表存储每个对象类型及关联类型的版本历史记录，每次对所述声明式映射配置的变更均以版本号递增方式写入所述本体模式注册表；所述映射引擎在读取历史数据时能够按照对应版本的映射配置执行转化，确保新旧数据的一致性及可追溯性。

说　明　书

一、技术领域

本发明属于数据工程与知识图谱技术领域，具体涉及一种将来自多个异构数据源的原始数据，通过统一适配器接口、声明式映射配置及本体事件总线，自动转化为语义本体图谱并支持语义查询的系统及方法。

二、背景技术

随着企业信息化建设的深入推进，大量业务数据分散存储于关系型数据库（MySQL、PostgreSQL）、文件系统（CSV、JSON）、REST API 等多种异构数据源中。这些数据之间缺乏统一的语义关联，形成"数据孤岛"（Data Silo）现象，给跨系统数据分析与决策支持带来严峻挑战。

现有技术存在以下缺陷：

缺陷一：集成耦合度高。传统 ETL 工具（如 Informatica、DataStage）采用点对点管道方式，N 个数据源需要 N×(N-1) 条集成链路，每新增一个数据源即需重新开发适配代码，运维成本呈指数级增长。

缺陷二：缺乏语义关联。现有数据仓库方案将数据以关系表形式存储，丢失了业务实体之间的语义关联关系，无法支持跨实体的关系路径查询（如"与某高风险供应商存在直接或间接合同关系的所有项目"）。

缺陷三：配置与代码耦合。数据源新增或字段变更时，需修改源代码、重新编译部署，响应周期长，无法满足业务敏捷性需求。

三、发明内容

3.1 技术问题

本发明所要解决的技术问题是：如何在不修改源代码的前提下，通过统一接口与声明式配置，将来自多个异构数据源的原始数据自动转化为带有语义关联的本体图谱，并支持高效的跨实体语义查询。

3.2 技术方案

为解决上述技术问题，本发明提供一种将异构原始数据转化为语义本体图谱的系统及方法。

其核心技术要点如下：

  • 设计统一适配器接口（SourceAdapter Trait），以标准化记录流（CanonicalRecord Stream）抽象所有数据源的输出格式，使数据源扩展与核心系统解耦；
  • 以声明式 TOML 映射配置文件描述数据源字段到本体对象类型属性的映射关系及外键关联关系，映射引擎在运行时动态加载配置，无需修改代码；
  • 映射引擎根据配置自动推导对象实例间的语义关联，生成本体事件（OntologyEvent）并写入事件总线（OntologyEventBus）；
  • 本体管理器（OntologyManager）订阅事件总线，将本体对象实例及语义关联持久化至本体知识库；
  • 提供面向对象集合（ObjectSet）的链式语义查询 API，支持关系路径导航、属性过滤及聚合计算。

3.3 有益效果

  • 降低集成复杂度：采用 Hub-and-Spoke 架构，将 N×(N-1) 条集成链路简化为 N 条适配器链路，显著降低运维成本；
  • 消除数据孤岛：通过语义关联推导，将离散数据源中的实体关系显式建模为语义本体图谱，支持跨实体关系路径查询；
  • 零代码扩展数据源：新增数据源只需新增一个实现 SourceAdapter 接口的适配器及对应的 TOML 配置文件，无需修改核心系统代码；
  • 事件驱动解耦：OntologyEventBus 实现数据采集与数据消费的异步解耦，便于多消费者并行处理（如实时查询、向量索引构建、审计日志）；
  • 支持 AI 语义查询：本体知识库结合向量索引，支持自然语言到结构化图查询的转化，为智能决策系统提供数据基础。

五、具体实施方式

5.1 实施方式一：统一适配器接口与声明式配置

本实施方式描述系统核心接口定义及声明式配置文件格式。

步骤一：定义统一适配器接口。系统定义 SourceAdapter Trait，规定所有适配器模块必须实现以下方法：

/// 统一适配器接口（Rust Trait 定义） pub trait SourceAdapter: Send + Sync { /// 适配器唯一标识符 fn source_id(&self) -> &str; /// 同步数据并输出标准化记录流 /// cursor: 增量游标，None 表示全量同步 fn ingest( &self, cursor: Option<Cursor>, ) -> impl Stream<Item = Result<CanonicalRecord>> + Send; } /// 标准化记录（每条原始数据记录的统一表示） pub struct CanonicalRecord { pub source_id : String, // 来源适配器标识 pub record_id : String, // 记录唯一标识 pub fields : HashMap<String, FieldValue>, // 字段键值对 pub ingested_at: DateTime<Utc>, // 采集时间戳 }

步骤二：实现文件适配器。以 CSV 文件适配器为例，具体实现如下：

pub struct FileAdapter { source_id : String, file_path : PathBuf, mapping : TomlMapping, // 从 TOML 配置文件加载的映射规则 } impl SourceAdapter for FileAdapter { fn source_id(&self) -> &str { &self.source_id } fn ingest( &self, _cursor: Option<Cursor>, ) -> impl Stream<Item = Result<CanonicalRecord>> + Send { let path = self.file_path.clone(); let mapping = self.mapping.clone(); async_stream::stream! { let mut rdr = csv::Reader::from_path(&path)?; for result in rdr.deserialize::<HashMap<String, String>>() { let row = result?; yield Ok(mapping.to_canonical(row)); // 按 TOML 规则转换 } } } }

步骤三：声明式映射配置示例。以下 TOML 配置描述了 CSV 文件到 Employee 对象类型的映射规则及外键关联推导：

[source] type = "csv" path = "data/employees.csv" [mapping] object_type = "Employee" primary_key = "id" [mapping.fields] name = "full_name" department = "dept" salary = "annual_salary" [mapping.foreign_keys] # 声明外键推导规则：字段 project_id → Employee -[WorksOn]→ Project project_id = { target_type = "Project", link_type = "WorksOn" }

上述 TOML 配置文件由映射引擎在运行时动态加载。当 CSV 文件中存在 project_id 字段时，映射引擎自动生成 WorksOn 类型的 LinkInstance，无需修改任何核心系统代码。

5.2 实施方式二：本体事件总线与语义查询

步骤四：本体事件定义。映射引擎将每条标准化记录转化为本体事件，写入事件总线：

/// 本体事件枚举（OntologyEvent） pub enum OntologyEvent { ObjectCreated(ObjectInstance), ObjectUpdated { id: ObjectId, diff: PropertyDiff }, ObjectDeleted(ObjectId), LinkCreated(LinkInstance), LinkDeleted { src: ObjectId, dst: ObjectId, link_type: LinkTypeId }, } /// 本体对象实例 pub struct ObjectInstance { pub id : ObjectId, pub type_id : ObjectTypeId, pub properties : HashMap<String, PropertyValue>, pub source_id : String, pub version : u64, } /// 语义关联实例 pub struct LinkInstance { pub src_id : ObjectId, pub dst_id : ObjectId, pub link_type : LinkTypeId, pub properties: HashMap<String, PropertyValue>, }

步骤五：本体管理器订阅事件总线并持久化。

impl OntologyManager { pub async fn run(&mut self) { while let Some(event) = self.event_bus.recv().await { match event { OntologyEvent::ObjectCreated(obj) => { self.store.upsert_object(obj).await?; } OntologyEvent::LinkCreated(link) => { self.store.upsert_link(link).await?; } OntologyEvent::ObjectDeleted(id) => { self.store.delete_object(id).await?; } _ => { /* 处理其余事件类型 */ } } } } }

步骤六：语义查询 API 链式调用。系统提供面向对象集合的链式查询接口，支持关系路径导航：

// 查询示例：获取所有工程部门员工所参与的高预算项目 let results = ontology .object_set("Employee") // 起始对象类型 .filter(|e| e["department"] == "Engineering") // 属性过滤 .navigate("WorksOn") // 沿 WorksOn 关系遍历 .filter(|p| p["budget"].as_f64() > 1_000_000.0) .collect::<Vec<ObjectInstance>>() .await?;

上述查询无需编写 SQL 或图查询语言，由系统底层将链式 API 调用转化为对本体知识库的遍历操作，返回满足条件的对象集合。

说　明　书　摘　要

本发明公开了一种将异构原始数据转化为语义本体图谱的系统及方法。所述方法通过统一适配器接口（SourceAdapter Trait）将来自本地文件系统、关系型数据库及 REST API 等多个异构数据源的原始数据转化为标准化记录流；依据声明式 TOML 映射配置，将所述标准化记录流转化为本体对象实例（ObjectInstance）及语义关联关系（LinkInstance），并通过外键字段自动推导对象间的语义关联，无需人工编写关联规则代码；所述本体对象实例及语义关联关系以事件流的形式写入本体事件总线（OntologyEventBus），并由本体管理器（OntologyManager）消费后持久化至本体知识库；系统还提供面向对象集合（ObjectSet）的链式语义查询 API，支持跨实体关系路径导航、属性过滤及聚合操作。

所述系统采用 Hub-and-Spoke 集成架构，将 N 个数据源的集成链路从 N×(N-1) 条减少为 N 条，显著降低数据集成运维成本；通过声明式配置与核心代码解耦，新增数据源仅需提供适配器实现与 TOML 配置文件，无需修改核心系统代码；本体事件总线支持多消费者并行处理，包括实时查询、向量索引构建及审计日志记录；系统进一步支持向量索引与图遍历的混合查询，为大型语言模型提供结构化数据检索能力，适用于企业级语义数据治理与 AI 增强决策平台。
