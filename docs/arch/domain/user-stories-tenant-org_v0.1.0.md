# Palantir — 租户 / 部门 / 项目管理 User Stories

> 版本：v0.1.0 | 日期：2026-03-19
> 关联：ADR-04 v2.0（三层模型）、user-roles_v0.2.0.md
>
> **三层模型回顾：**
> ```
> Tenant（公司）→ 强制隔离，不同企业绝对不互通
>   ├── Department（部门）→ 组织归属，默认数据可见性边界
>   └── Project（项目）→ 协作范围，跨部门，每人可参与多个
> ```
>
> **US 编号：**
> - `US-TN-xx` = Tenant 级别（企业注册、企业主管理）
> - `US-DP-xx` = Department 级别（部门管理）
> - `US-PJ-xx` = Project 级别（项目管理）

---

## Epic TN：Tenant 管理（企业主视角）

---

### US-TN-01：企业注册并创建第一个管理员

> P0 | 角色：新注册企业负责人 → 成为 TenantAdmin

**As** 企业负责人
**I want** 用企业邮箱注册一个新的企业账户，并自动成为该企业的管理员（TenantAdmin）
**So that** 我可以开始为公司搭建数据平台，管理员工账号和数据权限

**描述：**
注册时填写企业名称、行业、规模、管理员邮箱和密码。系统创建 Tenant 记录（`status = Trial`，默认 Standard 试用 14 天），自动创建第一个用户账号并赋予 `TenantAdmin` 角色，同时创建一个默认顶级部门（与企业同名）。注册成功后进入 Onboarding 向导。

**验收标准：**
- [ ] 企业名称 + 管理员邮箱组合唯一，重复时提示
- [ ] 注册完成后自动创建：1 个 Tenant + 1 个顶级 Department（企业名）+ 1 个 TenantAdmin 用户
- [ ] 试用期 14 天，到期前 3 天顶部显示到期提醒
- [ ] TenantAdmin 天然拥有 `is_tenant_admin = true`，可见全企业所有数据

**前端行为：**
- **入口：** `palantir.io/signup` — 企业注册页
- **表单：**
  - 企业名称（中英文均可）
  - 行业（下拉）+ 规模（下拉：10 人以下 / 10-100 / 100-1000 / 1000+）
  - 管理员邮箱 + 密码 + 确认密码
- 提交 → 邮件验证 → 激活链接跳转平台 → 触发 Onboarding 向导（US-SA-02）
- 激活后顶部显示企业名称 + 试用剩余天数徽章

**触发 API：** `POST /saas/tenants/register`

### US-TN-02：企业主查看全企业数据全景

> P0 | 角色：TenantAdmin

**As** TenantAdmin（企业主）
**I want** 在一个统一视图中看到全企业所有部门、所有项目的数据概览
**So that** 作为企业负责人，我对公司的数据资产有完整掌握，不受部门或项目边界限制

**描述：**
企业主的默认首页是"全企业总览"，展示：各部门数据量统计、活跃项目列表（含成员数和数据量）、近期数据变更、活跃用户排行。企业主可以直接穿透到任意部门或项目查看详情，无需申请任何权限。

**验收标准：**
- [ ] 展示企业下所有部门的数据对象数量（按 EntityType 分布）
- [ ] 展示所有项目（活跃 / 归档）及成员数量
- [ ] 近 7 天数据变更 Top10（哪些对象被频繁修改）
- [ ] 企业主访问任何部门 / 项目的数据时，不触发权限拒绝

**前端行为：**
- **入口：** 登录后默认首页（TenantAdmin 专属）
- **布局：**
  - 顶部：企业名 + Edition 徽章 + 总用户数 / 总对象数 / 活跃项目数（3 个数字卡）
  - 左侧：部门树（可展开，每个节点显示数据量）
  - 右侧主区：选中部门时展示该部门数据概览；选中项目时展示项目数据概览
  - 底部：近 7 天全企业 OntologyEvent 时间线（mini 版）
- 部门树节点右侧有"进入"按钮，点击切换主区视角
- 顶部搜索栏支持全企业跨部门/项目搜索

**触发 API：** `GET /v1/tenant/overview`

### US-TN-03：邀请成员加入企业

> P0 | 角色：TenantAdmin

**As** TenantAdmin
**I want** 通过邮件邀请同事加入我的企业租户，并为其分配初始 Persona 和所属部门
**So that** 公司员工都能使用平台，各自按职责访问数据

**描述：**
TenantAdmin 在"成员管理"页面批量输入邮箱（或上传 CSV），为每批成员选择 Persona 和初始部门。系统发送邀请邮件，被邀请人点击链接注册后自动加入对应部门。

**验收标准：**
- [ ] 支持单个邀请（输入邮箱）和批量邀请（上传 CSV：邮箱,Persona,部门）
- [ ] 邀请链接 72 小时有效，过期后 TenantAdmin 可重新发送
- [ ] 受邀人已有 Palantir 账号时直接加入（无需重新注册）
- [ ] 邀请成功后显示"待激活"状态，激活后变为"活跃"
- [ ] 当前成员数不超过 License 限制（超出时显示"已达上限，请升级 Edition"）

**前端行为：**
- **入口：** 管理控制台 → 成员管理 → "邀请成员"
- **Modal 表单（两种模式切换）：**
  - 单人模式：邮箱 + Persona 下拉 + 部门树选择器
  - 批量模式：CSV 上传区（模板下载按钮）+ 预览解析结果表格
- 提交后显示"已发送 N 封邀请，M 封失败（邮箱格式错误）"
- 成员列表有"待激活"筛选 Tab，可对待激活成员"重发邀请"

**触发 API：** `POST /v1/tenant/members/invite`

### US-TN-04：成员管理（Persona / 部门 / 停用）

> P0 | 角色：TenantAdmin

**As** TenantAdmin
**I want** 查看所有企业成员，修改其 Persona、所属部门，或在员工离职时停用账号
**So that** 企业人员变动时平台权限能及时同步

**描述：**
成员管理是 TenantAdmin 最常用的功能之一。支持搜索成员、批量修改、停用（不删除，保留操作历史）。停用时系统吊销该用户的所有 Token，并从所有项目中移除其 Editor/Owner 权限（保留 Viewer 历史记录）。

**验收标准：**
- [ ] 停用操作：立即吊销 Token（下次请求时 401），数据不删除
- [ ] 批量操作：可多选成员同时修改 Persona 或部门
- [ ] 修改 Persona / 部门后，EnrichedIdentity 缓存立即失效（NATS 事件驱动）
- [ ] 停用成员仍保留其历史操作的审计记录

**前端行为：**
- **入口：** 管理控制台 → 成员管理
- 表格列：头像 + 姓名 | 邮箱 | Persona | 所属部门 | 参与项目数 | 状态 | 最后活跃 | 操作
- 搜索栏：姓名 / 邮箱 / 部门过滤
- 行尾操作："编辑"（改 Persona/部门）/ "停用"（红色，二次确认）
- 多选后顶部浮现批量操作栏："修改 Persona" | "移动到部门" | "停用"

**触发 API：** `GET /v1/tenant/members`, `PUT /v1/tenant/members/{id}`, `POST /v1/tenant/members/{id}/deactivate`

## Epic DP：Department 管理

---

### US-DP-01：创建部门层级

> P0 | 角色：TenantAdmin

**As** TenantAdmin
**I want** 创建企业的部门树结构（如 总公司 → 市场部 → 华北区）
**So that** 组织架构在平台中有清晰映射，数据自动按部门归属进行权限隔离

**描述：**
部门以树形结构组织，支持多级（最多 5 级）。创建部门时可选择上级部门。部门创建后，归属该部门的员工的 OntologyObject 默认只有同部门（及其父部门 Admin）可见。

**验收标准：**
- [ ] 支持最多 5 级部门嵌套
- [ ] 部门名称在同级内唯一
- [ ] 删除部门前必须先移空成员（否则报错"该部门还有 N 名成员"）
- [ ] 创建后可立即为该部门指定 DeptAdmin

**前端行为：**
- **入口：** 管理控制台 → 组织架构
- **布局：** 左侧树形列表 + 右侧详情面板
  - 树节点：部门名 + 成员数 + 子部门数
  - 节点 Hover：出现 "+" 添加子部门 / "..." 更多操作
- 添加子部门 → 弹出小 Popover（输入名称 + 类型选择 + 描述）→ 确认即添加
- 拖拽节点可调整部门归属（有二次确认："将 N 名成员的部门归属一并迁移？"）
- 右侧详情：部门信息 + 成员列表 + DeptAdmin 设置 + 数据量统计

**触发 API：** `POST /v1/departments`, `PUT /v1/departments/{id}`, `GET /v1/departments/tree`

### US-DP-02：指定部门管理员（DeptAdmin）

> P0 | 角色：TenantAdmin

**As** TenantAdmin
**I want** 为每个部门指定一个或多个 DeptAdmin（部门长）
**So that** 部门长可以自主管理本部门的成员权限和项目，减少 TenantAdmin 的管理负担

**描述：**
DeptAdmin 是基于 ReBAC 的角色——用户被设置为某 OrgUnit 的 DeptAdmin 后，自动获得该部门（及其所有子部门）的数据查看权和成员管理权。DeptAdmin 可以：① 查看本部门所有项目和数据；② 将本部门成员加入项目；③ 修改本部门成员的 Persona。

**验收标准：**
- [ ] 一个部门可以有多个 DeptAdmin
- [ ] DeptAdmin 的权限范围：本部门 + 所有子部门（不能管理兄弟/父部门）
- [ ] 取消 DeptAdmin 权限后立即生效（EnrichedIdentity 缓存失效）

**前端行为：**
- **入口：** 组织架构 → 选中部门 → 右侧详情 → "部门管理员" 区块
- 显示当前 DeptAdmin 列表（头像 + 姓名）
- "添加管理员"按钮 → 成员搜索框（只搜索该部门成员）→ 选择 → 确认

**触发 API：** `POST /v1/departments/{id}/admins`

---

### US-DP-03：部门长查看本部门数据全景

> P0 | 角色：DeptAdmin

**As** DeptAdmin（部门长）
**I want** 查看我所在部门（及子部门）的所有数据和项目，不受具体项目成员限制
**So that** 作为部门负责人，我能掌握本部门所有的数据资产情况

**描述：**
DeptAdmin 登录后，首页展示其管辖部门的数据概览。与 TenantAdmin 不同，DeptAdmin 只能看到自己部门范围内的数据，看不到其他部门的项目和数据。

**验收标准：**
- [ ] DeptAdmin 能访问本部门所有项目的数据，无需是项目成员
- [ ] 不能访问其他部门（兄弟部门、父部门下其他子部门）的项目数据
- [ ] 本部门的 Agent 对话结果自动受部门数据边界约束

**前端行为：**
- **入口：** 登录后首页（DeptAdmin 专属视图）
- 布局与 TenantAdmin 类似，但部门树只显示自己管辖的部门
- 左侧树：本部门 + 子部门（灰色显示其他部门，不可点击）
- 顶部：部门名 + "共 N 名成员 / M 个活跃项目 / K 个数据对象"

**触发 API：** `GET /v1/dept/{id}/overview`

---

### US-DP-04：普通成员查看本部门共享数据

> P0 | 角色：普通成员（非 DeptAdmin，非 TenantAdmin）

**As** 普通部门成员（Analyst / Business User 等）
**I want** 自动能看到本部门的公共数据（不属于任何特定项目的共享数据），而不需要逐一申请权限
**So that** 新加入部门时，能立即访问到本部门的基础数据，快速上手

**描述：**
归属某部门的成员，自动获得该部门"共享数据"（`project_id = null` 且 `dept_id = 本部门`）的访问权，受 ABAC 部门过滤生效。对于有 `project_id` 的数据，只有显式加入对应项目才能看到。

**验收标准：**
- [ ] 新成员加入部门后（无需其他操作），能看到该部门的共享 OntologyObject
- [ ] 不能看到其他部门的共享数据
- [ ] 不能看到本部门的项目数据（除非显式加入项目）

**前端行为：**
- **数据查询页：** 默认展示"本部门共享数据"（无项目标签的对象）
- 数据列表左侧有"来源"列：显示"[部门共享]" 或 "[项目 X]"（只显示自己能访问的）
- Agent 对话结果只包含有访问权的数据

## Epic PJ：Project 管理

---

### US-PJ-01：创建项目

> P0 | 角色：任意 Persona（Analyst / App Builder / Data Scientist 等）

**As** 任意登录用户
**I want** 创建一个新项目，作为我和团队协作的数据空间
**So that** 项目内的数据和工作成果与其他部门/项目隔离，团队可以自由协作

**描述：**
任何用户都可以创建项目，创建者自动成为 Project Owner。创建时设置项目名称、描述、可见性（Private = 只有成员可见；DeptShared = 本部门成员可以 Discover；TenantShared = 全企业可以 Discover）。创建后可立即邀请成员。

**验收标准：**
- [ ] 创建者自动成为 Owner
- [ ] 项目名称在租户内唯一
- [ ] 可见性为 `DeptShared` / `TenantShared` 时，非成员可以看到项目存在（Discoverer 角色），但看不到数据
- [ ] 项目创建后显示在侧边栏"我的项目"列表中

**前端行为：**
- **入口：** 左侧侧边栏 → "项目" → "+" 新建项目
- **弹出 Modal：**
  - 项目名称（输入框 + 实时唯一性检测）
  - 描述（Textarea，可选）
  - 可见性选择（3 个 Radio 卡片，配图示说明）
  - 颜色/图标选择（个性化）
- 确认后 → 进入刚创建的项目首页，出现"邀请成员"引导提示

**触发 API：** `POST /v1/projects`

---

### US-PJ-02：邀请成员加入项目

> P0 | 角色：Project Owner / DeptAdmin（在自己部门范围内）

**As** Project Owner
**I want** 邀请同事加入我的项目，并为每人分配 Owner / Editor / Viewer 角色
**So that** 团队成员可以在项目内协作，各自有合适的操作权限

**描述：**
项目成员可以跨部门（只要是同一 Tenant 的成员）。添加成员时选择权限等级：Owner（可管理项目）/ Editor（可读写数据）/ Viewer（只读）。被添加的成员立即能在"我的项目"中看到该项目并访问项目数据。

**验收标准：**
- [ ] 只能邀请同 Tenant 内的成员（不能跨企业）
- [ ] 添加成员后立即生效（不需要被邀请人确认）
- [ ] 同一成员可以在不同项目有不同角色（项目 X Editor，项目 Y Viewer）
- [ ] Project 只能有 1 个 Owner，转让 Owner 时原 Owner 降为 Editor

**前端行为：**
- **入口：** 项目首页 → 右上角"成员"图标 → 成员面板
- 面板顶部：搜索框（搜索企业成员姓名/邮箱）
- 搜索结果：成员卡片（头像 + 姓名 + 所属部门）
- 点击"邀请"→ 角色下拉（Owner/Editor/Viewer）→ 确认
- 成员列表：可修改角色 / 移除成员（Owner 不能移除自己）

**触发 API：** `POST /v1/projects/{id}/members`, `PUT /v1/projects/{id}/members/{user_id}`

---

### US-PJ-03：项目成员查看项目数据

> P0 | 角色：ProjectEditor / ProjectViewer

**As** 项目成员
**I want** 在项目视图中查看该项目内的所有数据，就像进入一个专属工作空间
**So that** 项目工作专注在项目数据上，不受无关数据干扰

**描述：**
进入项目后，所有数据操作（查询、Agent 对话、创建对象）都默认在项目范围内。左侧侧边栏切换项目时，主区数据视图随之切换。Agent 对话也自动限制在当前项目的数据范围内回答。

**验收标准：**
- [ ] 项目视图下，查询结果只包含 `project_id = 当前项目` 的对象
- [ ] Agent 对话在项目视图下，只基于项目数据回答
- [ ] 创建新对象时，默认 `project_id = 当前项目`
- [ ] 项目 Viewer 不能创建/修改对象（操作按钮置灰 + Tooltip）

**前端行为：**
- **布局：** 左侧侧边栏有"项目"分区，列出"我的项目"列表
  - 选中项目 → 顶部面包屑更新为 "企业名 / 项目名"
  - 主区所有内容都是该项目数据
- 项目徽章：数据对象列表每行都有项目标签（颜色与项目颜色对应）
- 顶部下拉切换："全部我的数据" | "部门共享" | "项目 X" | "项目 Y"

### US-PJ-04：申请加入项目（Discoverer → 申请 Viewer）

> P1 | 角色：能 Discover 到项目的任意成员

**As** 能看到项目存在（Discoverer）但没有加入的成员
**I want** 看到项目名称和简介，并能申请加入
**So that** 感兴趣的项目可以主动申请参与，而不是等待被邀请

**描述：**
可见性为 `DeptShared` 或 `TenantShared` 的项目对符合条件的成员可见（但数据不可见）。这些成员可以点击"申请加入"，填写申请理由，通知 Project Owner 审批。审批通过后成员获得 Viewer 权限（Owner 可手动调整为 Editor）。

**验收标准：**
- [ ] Discoverer 能看到项目名称、描述、Owner、成员数，但不能看数据
- [ ] 申请后 Owner 收到站内通知 + 邮件
- [ ] Owner 可以批准（选择赋予 Viewer/Editor）或拒绝（填写原因）
- [ ] 申请人收到审批结果通知

**前端行为：**
- **入口：** 侧边栏 → "发现项目"（仅展示 DeptShared/TenantShared 项目）
- 项目卡片：项目名 + Owner 头像 + 描述摘要 + "申请加入"按钮
- 点击"申请加入"→ 小 Modal（申请理由文本框）→ 提交
- Owner 的通知中心：新申请提醒 → 点击进入审批 → 批准/拒绝

**触发 API：** `POST /v1/projects/{id}/join-requests`, `PUT /v1/projects/{id}/join-requests/{req_id}`

---

### US-PJ-05：将数据归属到项目

> P0 | 角色：ProjectEditor / Data Engineer

**As** 有写权限的项目成员
**I want** 在创建 OntologyObject 时，将其归属到当前项目，或将已有的共享数据移入项目
**So that** 项目数据有清晰的归属边界，只有项目成员能访问

**描述：**
两种场景：① 在项目视图内创建对象 → 对象自动带 `project_id`；② 将部门共享数据（`project_id = null`）明确归入某项目（需要数据 Owner 权限）。归属后原来能看这条数据的成员（部门过滤）将失去访问权，只有项目成员能访问。

**验收标准：**
- [ ] 在项目视图创建的对象默认带 `project_id = 当前项目`
- [ ] 将共享数据移入项目时，有警告："以下 N 个有权访问此数据的部门成员将失去访问权"
- [ ] 数据归属变更写入审计日志
- [ ] 可以将数据从项目移回共享（需要 Project Owner 或 TenantAdmin）

**前端行为：**
- **入口：** 数据对象详情页 → "归属" 区块（显示当前归属：部门共享 / 项目名）
- "修改归属"按钮 → 项目选择器下拉（只显示用户有 Editor 及以上权限的项目）
- 确认前展示影响说明 Dialog："将影响 N 名当前有权访问的成员"
- 批量操作：数据列表页多选 → "移动到项目"

**触发 API：** `PUT /v1/objects/{id}/project`

### US-PJ-06：普通成员在多个项目间切换

> P0 | 角色：任意项目成员

**As** 同时参与多个项目的成员（如 Analyst 在项目 X 是 Editor，项目 Y 是 Viewer）
**I want** 在不同项目间快速切换，清楚知道当前处于哪个项目的数据视角
**So that** 多项目工作时不会混淆数据，每个项目的 Agent 对话和数据查询相互独立

**描述：**
左侧侧边栏固定展示"我的项目"列表，点击切换当前工作项目。切换项目后，当前 Agent 会话的数据范围跟着变化（或提示"开启新会话以切换项目数据范围"）。面包屑始终显示当前项目名称。

**验收标准：**
- [ ] 侧边栏"我的项目"列表按最近访问排序，最多显示 10 个，其余折叠
- [ ] 切换项目后数据查询、Agent 对话自动限定在新项目范围
- [ ] 当前项目在侧边栏高亮，面包屑显示"企业 / 部门 / 项目名"
- [ ] 在 Viewer 角色的项目中，创建/修改按钮自动置灰

**前端行为：**
- **侧边栏结构（固定）：**
  ```
  🏢 企业名
  ├── 📁 我的数据（部门共享）
  ├── 📂 我的项目
  │   ├── ● 项目 X（Editor）← 当前高亮
  │   ├── ○ 项目 Y（Viewer）
  │   └── ○ 项目 Z（Owner）
  └── 🔍 发现更多项目
  ```
- 点击切换：主区立刻刷新，面包屑更新
- Viewer 项目：项目名旁显示"只读"徽章，主区操作按钮置灰
